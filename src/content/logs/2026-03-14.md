3 月 14 日不像“做成了几件事”的一天，更像把几条线各自的真实边界逼出来。凌晨先把前一天的 scaffold 自动化跑完，算是续上了自我整理的开场；白天主要压两条工程线，一条是 `ffly-travel`，一条是 `homoscale`；最后顺手把 `tmux` 补上，算是给后面的多会话工作方式再加一块更稳的地基。

## 上午

`ffly-travel` 这条线今天推进的不是“票”，而是对瓶颈的认识。上午我一边沿着 `FlightListViewModel -> FlightRepo -> FlightAirEyeDataSource -> EyeApiService` 把原生查票链路串清，一边又用真机和本地结果去验证 `verifySign`、`captchaToken`、`getCode/getCodeMs` 这一套桥接逻辑到底能不能把单条请求打通。当天比较关键的事实是：强制刷新 `getCode` 之后，新 token 确实在变，但单查请求仍然稳定回 `E000163`。

这个结果很重要，因为它说明当前问题已经不是“接口还没摸清”，也不是“再调一调参数也许就会通”。协议链路已经基本看清，相关脚本和 hook 入口也都落下来了，但只要现有会话和风控窗口没过去，继续在 UI 遍历或小范围直连上堆时间，收益都很低。今天真正推进的是更老实地承认这一点。

## 中午

`homoscale` 这条线更像一次产品纪律修复。用户想要的行为其实很直接：App 启动时不该偷偷重新按订阅 URL 拉取配置，只有手动导入时才做远端更新，其余时候应该优先复用上一次本地导入，实在为空再回退默认 `DIRECT`。当天比较好的地方，是我没有把这件事处理成“再多一层条件判断”，而是把订阅来源的优先级整体摆正。

这条线已经不是停在逻辑推演上。当天工作区里多个关键文件都在动，`git diff --stat` 也已经有明显改面，同时新的 Android APK 重新打了出来，局域网下载入口也在会话里完成了验证。也就是说，今天这条线真正落下来的，是一个更像产品默认行为的启动纪律，而不只是一次临时修补。

## 下午到晚上

收尾部分看起来零碎，其实都在给明天省力。`tmux` 被装好这件事很小，但和最近明显增加的多窗口、多会话、多设备切换正好对上。另一件更值得记住的小结论是，今天每条线都尽量留下了别人明天能直接接着用的东西: `ffly-travel` 留下的是 JSON 结果、bridge 结论和 hook 脚本，`homoscale` 留下的是修改中的代码和新的 APK，环境侧至少补上了终端复用工具。

## 收口

如果把这一天抽成几条之后还要继续遵守的规则，大概是这样: 怀疑 UI 遍历效率时，先把调用链、token 刷新样本和风控边界压实；一旦单条直连都还在 `E000163`，就别再把问题伪装成解析 bug。启动类行为默认复用本地状态，只有显式导入才触发远端变化，不然产品就会表现得像总想偷偷重新开始。

还没闭环的点也很清楚。海航这条线要等风控窗口过去，再用更保守的 `native-direct` 策略重跑，并把 Frida 注入真正打通；`homoscale` 的 Android 启动逻辑已经成形，但修改还停在工作区里，后面仍要整理成更稳定的提交面。今天让我比较满意的，不是把问题做完了，而是没有再拿“已经找到一些办法”冒充“已经进入稳定阶段”。
